## 4. New Bugs Found in R3

### BUG-015: `activeBlockIdRef` Not Cleared When Session Fails to Create

**Severity**: Medium
**Location**: `C:\Velocity\src\components\Terminal.tsx`, lines 95-101

**Description**: In the `startSession` function's `catch` block:
```tsx
catch (err) {
  const errorBlock = createBlock('', shell);
  errorBlock.output = `[Failed to create session: ${err}]`;
  errorBlock.status = 'completed';
  activeBlockIdRef.current = errorBlock.id;
  setBlocks([errorBlock]);
}
```

The error block is created with `status: 'completed'`, but `activeBlockIdRef.current` is set to this error block's ID. Since the session was never created, `sessionIdRef.current` remains `null` (or whatever it was before). The input field is not disabled because `closed` is `false` (only `setClosed(true)` via `pty:closed` event sets it). No event listeners are registered.

If the user types a command and presses Enter after a failed session creation:
1. `submitCommand` checks `!sessionIdRef.current || closed` -- `sessionIdRef.current` is `null`, so the function returns early
2. The command is effectively ignored

This is safe behavior (no crash), but the UX is confusing: the input field appears active and accepts typing, but Enter does nothing. The user has no indication that the session failed unless they notice the error text in the error block.

**Impact**: Confusing UX after session creation failure. User can type but nothing happens.

**Recommendation**: Either set `setClosed(true)` in the catch block to show the Restart button, or disable the input field when there is no active session.

### BUG-017: Welcome Block Can Be "Rerun" with Empty Command

**Severity**: Low
**Location**: `C:\Velocity\src\components\blocks\BlockView.tsx`, lines 60-70

**Description**: The welcome block correctly hides the "Copy Command" and "Rerun" buttons via the `isWelcome` check:
```tsx
const isWelcome = block.command === '';
...
{!isWelcome && (
  <>
    <button ...>Copy Command</button>
    <button ...>Rerun</button>
  </>
)}
```

However, the "Copy Output" button is always shown (even for welcome blocks). This is intentional and correct -- users should be able to copy the shell banner/prompt output.

But if a non-welcome block were created with an empty command string (e.g., by pressing Enter with empty input), the block would be treated as a welcome block:
- No header (command text, timestamp) shown
- No Copy Command or Rerun buttons

Looking at `submitCommand`:
```tsx
const submitCommand = useCallback(
  (command: string) => {
    if (!sessionIdRef.current || closed) return;
    const newBlock = createBlock(command, shellType);
    ...
```

There is no guard against empty commands. If the user presses Enter with empty input:
```tsx
const handleKeyDown = useCallback(
  (e: React.KeyboardEvent<HTMLInputElement>) => {
    if (e.key === 'Enter' && sessionIdRef.current && !closed) {
      submitCommand(input);
      setInput('');
    }
  },
```

An empty string `''` is passed to `submitCommand`, which creates a new block with `command: ''`. This block:
1. Is created and added to the blocks array
2. The previous active block is marked as `completed`
3. The empty command `'' + '\r'` is written to the PTY (just a carriage return)
4. The new block renders as a welcome block (no header, no Copy Command, no Rerun)

**Impact**: Pressing Enter with empty input creates a block that looks like a welcome block. The PTY receives a bare `\r`, which in PowerShell/CMD produces a new prompt line. The output goes into this "welcome-like" block. This is cosmetically wrong -- a user-submitted block should show a header (even if the command is empty) -- but it is not functionally broken.

**Recommendation**: Either prevent empty command submission (guard `if (!command.trim()) return;` in `submitCommand`), or use a dedicated `isWelcome` flag on the Block type instead of checking `command === ''`.

### BUG-019: `activeBlockIdRef` Stale Reference After Block Eviction by MAX_BLOCKS

**Severity**: **High**
**Location**: `C:\Velocity\src\components\Terminal.tsx`, lines 164-191

**Description**: When `submitCommand` is called, the following sequence occurs:
```tsx
setBlocks((prev) => {
  const updated = prev.map((b) =>
    b.id === activeBlockIdRef.current
      ? { ...b, status: 'completed' as const }
      : b,
  );
  const withNew = [...updated, newBlock];
  return withNew.length > MAX_BLOCKS
    ? withNew.slice(-MAX_BLOCKS)
    : withNew;
});
activeBlockIdRef.current = newBlock.id;
```

The new block is always appended at the end and `slice(-MAX_BLOCKS)` keeps the last N blocks, so the new block is always in the kept set. `activeBlockIdRef.current` is set to `newBlock.id` which is in the array. This is correct.

However, consider the `pty:output` event listener:
```tsx
setBlocks((prev) =>
  prev.map((b) =>
    b.id === activeBlockIdRef.current
      ? { ...b, output: b.output + event.payload }
      : b,
  ),
);
```

This maps over ALL blocks looking for `activeBlockIdRef.current`. Since the active block is always the newest one (appended at the end), and `slice(-MAX_BLOCKS)` keeps the last N, the active block is never evicted. **This is correct.**

But there is a performance concern: every PTY output event causes a `prev.map()` over all blocks (up to 50). Each map creates a new array with new block objects for the matching block. For high-throughput PTY output (e.g., `dir /s`), this means:
- Hundreds of `setBlocks` calls per second
- Each creates a new array of up to 50 blocks
- Each triggers React reconciliation over all BlockView components
- `React.memo` on BlockView helps (unchanged blocks get referential equality check), but the active block changes every time, causing at least 1 BlockView re-render per event
- `AnsiOutput` inside the active block uses `useMemo(() => parseAnsi(text), [text])`, but `text` changes every event, so `parseAnsi` re-runs every time on the ENTIRE accumulated output

**Assessment**: This is the same performance concern as R1 BUG-004 but now amplified by the block model. Previously it was a single string buffer re-parsed every event. Now it is a single block's output string re-parsed every event, plus React reconciliation over the blocks array.

The performance impact is identical to the pre-block-model behavior for the active block, but the block array mapping adds overhead. For 50 blocks, the map + spread is negligible compared to the `parseAnsi` re-parse.

**Impact**: Performance degradation under heavy PTY output. Not a correctness bug.

**Recommendation**: This was already noted as R1 BUG-004. Consider appending to a list of output chunks instead of concatenating strings, and rendering output incrementally.

### BUG-020: Welcome Block Status Never Set to 'completed'

**Severity**: Medium
**Location**: `C:\Velocity\src\components\Terminal.tsx`, lines 56-59, 168-173

**Description**: When a session starts, a welcome block is created:
```tsx
const welcomeBlock = createBlock('', shell);
activeBlockIdRef.current = welcomeBlock.id;
setBlocks([welcomeBlock]);
```

The welcome block starts with `status: 'running'`. When the user submits their first command, `submitCommand` runs:
```tsx
setBlocks((prev) => {
  const updated = prev.map((b) =>
    b.id === activeBlockIdRef.current
      ? { ...b, status: 'completed' as const }
      : b,
  );
  const withNew = [...updated, newBlock];
  ...
});
activeBlockIdRef.current = newBlock.id;
```

This correctly sets the welcome block to `completed` when the first command is submitted. **But what if the user never submits a command?** The welcome block stays as `status: 'running'` forever. If the process exits without the user ever typing:
1. The `pty:closed` event fires
2. `setClosed(true)` runs
3. The welcome block still has `status: 'running'`
4. The BlockView renders: `isActive && block.status === 'running'` -> the running indicator (pulsing dot) shows forever next to a block with no command

Wait -- the welcome block has `command === ''`, so `isWelcome` is true, and the header (which contains the running indicator) is NOT rendered for welcome blocks. So the running indicator does NOT show for the welcome block. But the `block-active` CSS class IS applied:
```tsx
className={`block-container ${isActive && block.status === 'running' ? 'block-active' : ''}`}
```

So the welcome block gets the blue left border (`block-active`) even after the process exits. This is a minor visual inconsistency -- a closed session's welcome block still appears "active".

**Impact**: Minor visual issue. The welcome block retains a blue left border indicator after the session closes, if the user never typed a command.

**Recommendation**: When the `pty:closed` event fires, mark the current active block as `completed`:
```tsx
setBlocks((prev) =>
  prev.map((b) =>
    b.id === activeBlockIdRef.current
      ? { ...b, status: 'completed' as const }
      : b,
  ),
);
```

### BUG-021: `handleCopyOutput` Uses Stale `block.output` Due to `useCallback` Dependency

**Severity**: Medium
**Location**: `C:\Velocity\src\components\blocks\BlockView.tsx`, lines 23-27

**Description**: The `handleCopyOutput` callback:
```tsx
const handleCopyOutput = useCallback(() => {
  navigator.clipboard.writeText(stripAnsi(block.output)).catch(() => {});
}, [block.output]);
```

This is wrapped in `useCallback` with `[block.output]` as the dependency. When the user clicks "Copy Output" on the active block while output is still streaming, the callback captures the `block.output` value at the time of the most recent render.

Due to `React.memo` on BlockView and the fact that the block object reference changes on every PTY output event (because `setBlocks` creates new block objects via spread), this re-render happens on every output event for the active block. So the callback is actually up-to-date on every render.

**Assessment**: Not actually stale -- `React.memo` compares props shallowly, and since the block object is new on every output event, BlockView re-renders and the callback gets a fresh closure. The `useCallback` is effectively a no-op in terms of preventing re-creation since the dependency changes every time.

**Impact**: None. The concern is theoretical and does not manifest.

**Note**: The `useCallback` on `handleCopyOutput` provides no memoization benefit for the active block since `block.output` changes on every event. It does provide benefit for completed blocks where `block.output` is stable. This is acceptable.

### BUG-022: Rapid Shell Switching Race Condition Now Creates Orphaned Blocks

**Severity**: Medium
**Location**: `C:\Velocity\src\components\Terminal.tsx`, lines 106-120

**Description**: This is an extension of R2 BUG-009 in the context of blocks. When `resetAndStart` is called:
```tsx
const resetAndStart = useCallback(
  async (shell: ShellType) => {
    if (sessionIdRef.current) {
      await closeSession(sessionIdRef.current).catch(() => {});
    }
    cleanupListeners();
    setBlocks([]);
    activeBlockIdRef.current = null;
    setInput('');
    setClosed(false);
    updateSessionId(null);
    await startSession(shell);
  },
  [cleanupListeners, startSession, updateSessionId],
);
```

If two concurrent `resetAndStart` calls execute (from rapid shell switching per BUG-009):
1. Call A: `setBlocks([])` -> clears blocks
2. Call A: `await startSession(shellA)` -> creates welcome block, sets `activeBlockIdRef` to welcome block A
3. Call B (before A completes): `setBlocks([])` -> clears blocks including welcome block A
4. Call B: `await startSession(shellB)` -> creates welcome block B, sets `activeBlockIdRef` to welcome block B

The problem: between step 2 and step 3, PTY output from session A goes into welcome block A via the output listener. Then step 3 clears all blocks. The listener for session A is cleaned up by step 3's `cleanupListeners()` call (in the second `resetAndStart` call).

But wait: `cleanupListeners()` removes the listeners registered in step 2's `startSession`. So after step 3, no listeners exist for session A. Then step 4 creates new listeners for session B.

Session A's backend process is still running (step 3's `closeSession` call in the second invocation would close whatever `sessionIdRef.current` points to, which is session A's ID by step 2). So session A IS closed.

Actually, let me re-trace more carefully:
1. First `resetAndStart` begins:
   - `closeSession(old-session)` -- close original session
   - `cleanupListeners()` -- remove original listeners
   - `setBlocks([])`, `activeBlockIdRef = null`, etc.
   - `await startSession(shellA)` -- creates session A, sets `sessionIdRef` to A, registers A's listeners, creates welcome block A
2. Second `resetAndStart` begins (before first fully resolves from React perspective):
   - `closeSession(sessionIdRef.current)` -- closes session A (from step 1)
   - `cleanupListeners()` -- removes session A's listeners (from step 1)
   - `setBlocks([])` -- clears welcome block A
   - `activeBlockIdRef = null`
   - `await startSession(shellB)` -- creates session B

This is identical to R2 BUG-009 but with blocks. The effect is the same: orphaned session + resource leak. The block model does not introduce new bugs here, just a new dimension to the same existing bug.

**Impact**: Same as BUG-009. Orphaned PTY sessions on rapid shell switching.

**Recommendation**: Same as BUG-009. Add re-entry guard.

### BUG-025: Output Buffer Has No Size Limit Per Block

**Severity**: Medium
**Location**: `C:\Velocity\src\components\Terminal.tsx`, lines 64-68

**Description**: In the R2 codebase, there was an `OUTPUT_BUFFER_LIMIT` of 100K characters that capped the total output string. The Block Model replaces the single output string with per-block output strings. However, there is NO per-block output limit:

```tsx
const unlistenOutput = await listen<string>(
  `pty:output:${sid}`,
  (event) => {
    setBlocks((prev) =>
      prev.map((b) =>
        b.id === activeBlockIdRef.current
          ? { ...b, output: b.output + event.payload }
          : b,
      ),
    );
  },
);
```

Each block's `output` string grows unboundedly. A single long-running command (e.g., `Get-ChildItem C:\ -Recurse`) could accumulate megabytes of output in a single block.

The previous `OUTPUT_BUFFER_LIMIT` of 100K was removed (or never added to the block model). The MAX_BLOCKS limit of 50 only limits the NUMBER of blocks, not the SIZE of each block's output.

**Impact**: Memory growth under heavy output. A single block could consume excessive memory and cause the AnsiOutput component to become extremely slow (re-parsing megabytes of ANSI text on every update).

**Recommendation**: Add a per-block output limit (e.g., 100KB per block), or implement output virtualization (only render visible output lines).

## 5. Security Assessment

### Positive Findings

1. **No `unwrap()` on user-derived data** -- All Rust code uses `map_err()` or `unwrap_or()`.
2. **ANSI filter comprehensive** -- Deny-by-default, only SGR passes through.
3. **Block output rendered via AnsiOutput -> React JSX** -- Auto-escapes HTML, no XSS risk.
4. **Shell type validation** -- `validate_shell_type()` restricts to known shells.
5. **Session ID as UUID v4** -- Cannot be guessed.
6. **Copy Output strips ANSI** -- `handleCopyOutput` calls `stripAnsi(block.output)` before clipboard write, preventing ANSI sequences from leaking to the clipboard.
7. **Block model does not introduce new IPC surface** -- No new Tauri commands were added for blocks. All block logic is frontend-only.

### Security Concern: Block Output Contains Unescaped Error Interpolation

From R2 BUG-014, error messages are interpolated into block output:
```tsx
{ ...b, output: b.output + `\n[Error: ${event.payload}]\n` }
{ ...b, output: b.output + `\n[Write error: ${err}]\n` }
```

These are then rendered through `AnsiOutput`, which passes them through `Anser.ansiToJson()`. If the error payload contained ANSI sequences, they would be styled. This is not exploitable since Rust error strings do not contain ANSI, but the pattern is noted.

## 9. Overall Assessment

**Verdict**: PASS with notes

The Block Model is well-structured with clean separation between data (Block type), presentation (BlockView), and state management (Terminal). The code is readable, follows React best practices (useCallback, useMemo, React.memo), and the test coverage for the new components is solid.

**Strengths:**
- Clean `Block` type definition with appropriate fields
- `createBlock` factory function ensures consistent block creation
- `activeBlockIdRef` pattern correctly routes PTY output to the current block
- Welcome block distinguished by `command === ''` -- simple and effective
- `React.memo` on BlockView and `useMemo` on AnsiOutput provide render optimization
- Copy Output correctly strips ANSI via `stripAnsi()` before clipboard write
- MAX_BLOCKS limit (50) prevents unbounded block accumulation
- Block actions (Copy Command, Copy Output, Rerun) are hidden via CSS hover -- clean UX
- Shell switch and restart correctly clear blocks via `resetAndStart`
- 14 new tests covering block data model, BlockView component, and terminal integration

**Primary Concerns:**
1. **BUG-025 (Medium)**: Per-block output has no size limit. The previous 100K global buffer limit was lost in the transition to blocks. A single long-running command can cause unbounded memory growth.
2. **BUG-020 (Medium)**: Welcome block retains active styling after process exit (easy fix).
3. **BUG-015 (Medium)**: Confusing UX after session creation failure (input appears active but does nothing).
4. **BUG-009 (Medium, from R2)**: Rapid shell switching race condition still open.

**No critical or high-severity bugs found.**

The block model is a solid foundation for the Velocity terminal. The bugs found are medium and low severity, with straightforward fixes. The test coverage has grown significantly (+14 tests, +2 test files) but there are notable gaps in integration testing (output accumulation, status transitions, MAX_BLOCKS enforcement behavior) that should be addressed in the next iteration.
