## 4. New Bugs Found

### BUG-026: Multi-line Command Sent As Single Write With Only One `\r` Suffix

**Severity**: Medium
**Location**: `C:\Velocity\src\components\Terminal.tsx`, line 237

**Description**: When the user creates a multi-line input using Shift+Enter (e.g., `echo line1\necho line2`) and presses Enter to submit, the `submitCommand` function writes:
```typescript
writeToSession(sessionIdRef.current, command + '\r')
```

This sends the entire multi-line string with a single carriage return at the end: `"echo line1\necho line2\r"`. The newline character `\n` within the text is NOT converted to `\r` (which is what a terminal expects for line execution). The behavior depends on the shell:

- **PowerShell**: May interpret the `\n` as a literal newline within a multi-line statement rather than executing each line separately. In PowerShell, multi-line input requires specific syntax (e.g., backtick continuation, or pasting with Ctrl+V which sends individual keystrokes).
- **CMD**: `\n` is not a valid line separator in CMD's PTY input; only `\r` triggers command execution.

**Impact**: Multi-line input created via Shift+Enter may not execute as the user expects. The user types two separate commands on two lines but only the second line executes (or the whole thing is treated as one mangled command).

**Recommendation**: Convert `\n` characters in multi-line input to `\r` before sending to the PTY, so each line is executed as a separate command: `command.replace(/\n/g, '\r') + '\r'`.

### BUG-027: Tab Key Cursor Position Restoration May Fail Under React Batching

**Severity**: Low
**Location**: `C:\Velocity\src\components\editor\InputEditor.tsx`, lines 36-39

**Description**: The Tab handler inserts two spaces and then uses `requestAnimationFrame` to restore the cursor position:
```typescript
requestAnimationFrame(() => {
  textarea.selectionStart = start + 2;
  textarea.selectionEnd = start + 2;
});
```

React 18's automatic batching may cause the `onChange` state update to not be reflected in the DOM by the time `requestAnimationFrame` fires. Specifically:
1. `onChange(newValue)` enqueues a state update
2. `requestAnimationFrame` fires before React flushes the state update to the DOM
3. The textarea's value is still the old value, so setting `selectionStart`/`selectionEnd` has no effect (or the position is clamped to the old string length)

In practice, React typically flushes state updates before the next paint (which is when rAF fires), so this usually works. But under heavy load or when React batches multiple updates, the timing may be off.

**Impact**: Cursor may jump to the wrong position after Tab insertion in some edge cases. Difficult to reproduce in normal use.

**Recommendation**: Use a `useEffect` that runs after the state update and the DOM re-render, rather than `requestAnimationFrame`. Alternatively, use `flushSync` from `react-dom` to force synchronous update before setting cursor position.

### BUG-028: Tokenizer Misclassifies Flag-Like Arguments After Redirect

**Severity**: Low
**Location**: `C:\Velocity\src\lib\shell-tokenizer.ts`, lines 67-75

**Description**: Consider the input: `echo hello > -output.txt`. The `>` sets `expectCommand = false`, so the next word `-output.txt` is correctly NOT classified as a command. However, the code then checks:
```typescript
} else if (word.startsWith('--') || word.startsWith('-')) {
  tokens.push({ type: 'flag', value: word });
}
```

Since `-output.txt` starts with `-`, it is classified as a `flag` rather than an `argument`. This is a display-only issue (the tokenizer is not used for execution), but it means a filename starting with `-` after a redirect gets highlighted in yellow (flag color) instead of the default text color.

**Impact**: Minor syntax highlighting inaccuracy for edge case filenames beginning with `-` after redirects. Visual-only.

**Recommendation**: After a redirect operator (`>`, `>>`, `<`), the next non-whitespace token should always be classified as `argument` regardless of whether it starts with `-`. This would require tracking a `expectRedirectTarget` state.

### BUG-029: Tokenizer Does Not Handle Semicolons or `&&` / `||` as Command Separators

**Severity**: Low
**Location**: `C:\Velocity\src\lib\shell-tokenizer.ts`, line 32

**Description**: The tokenizer's regex does not recognize `;`, `&&`, or `||` as command separators. For input like:
- `echo hello; echo world` -- `echo world` should be `command argument`, but `echo` is classified as `argument` because `;` is not a known separator
- `ls && echo done` -- `echo` should be a command after `&&`, but is classified as `argument`

The current regex pattern only recognizes `|`, `>`, `>>`, `<` as special operators.

**Impact**: Display-only. Commands after `;`, `&&`, `||` are not highlighted as commands (shown in default text color instead of blue). The commands still execute correctly because the tokenizer is visual-only.

**Recommendation**: Add `;`, `&&`, and `||` to the pipe/operator regex group and set `expectCommand = true` after them. This is a future improvement, not a blocking issue.

### BUG-030: Overlay `<pre>` Height May Not Match Textarea Height

**Severity**: Medium
**Location**: `C:\Velocity\src\components\editor\InputEditor.tsx`, lines 50-57 and `C:\Velocity\src\App.css`, lines 223-255

**Description**: The overlay technique relies on pixel-perfect alignment between the `<pre>` (highlight layer) and `<textarea>` (input layer). Both have matching CSS for font, font-size, line-height, padding, margin, and white-space. However:

1. **Height mismatch**: The textarea uses `rows={lineCount}` to set its height, but the `<pre>` has no explicit height -- it sizes to its content. If the textarea's computed height from `rows` differs from the `<pre>`'s content height (e.g., due to browser-specific textarea line-height computation), the highlight can shift vertically.

2. **Border/padding differences**: Browsers add default `padding` and `border` to `<textarea>` elements that may not be fully reset by `padding: 0; border: none`. Some browsers add `2px` of internal padding. The CSS explicitly sets `padding: 0; margin: 0; border: none` which should work, but the textarea's intrinsic sizing from `rows` may not match the pre's content height exactly.

3. **Trailing newline hack**: The `<pre>` includes a trailing `{'\n'}` (line 56) to match the textarea's dimensions. This is a known technique, but if the textarea has a different trailing space behavior, the last line may misalign.

**Impact**: On some browsers or at certain font sizes, the highlighted text may drift 1-2 pixels from the actual text cursor position. This is a cosmetic issue that becomes more noticeable on multi-line input.

**Recommendation**: Instead of using `rows={lineCount}`, compute both elements' heights from the same source. One approach: let the `<pre>` define the height and set the textarea to `height: 100%` of the container, or use a ResizeObserver to sync heights.

### BUG-032: `disabled` Prop Does Not Prevent Keyboard Shortcuts in InputEditor

**Severity**: Low
**Location**: `C:\Velocity\src\components\editor\InputEditor.tsx`, lines 21-44

**Description**: The `handleKeyDown` callback does not check the `disabled` prop before processing keyboard events. While the HTML `disabled` attribute on the textarea prevents the browser from accepting input (onChange will not fire), the `onKeyDown` event DOES fire on disabled textareas in some browsers.

If `onKeyDown` fires while disabled:
- **Enter**: Would call `onSubmit(value)` with the current value. Since `submitCommand` checks `!sessionIdRef.current || closed`, this is safe (the submit would be a no-op).
- **Tab**: Would call `onChange(newValue)` with spaces inserted. Since the textarea is disabled, the React state update may succeed but the textarea won't reflect it.

**Impact**: Minimal. The parent component's guards prevent actual damage. But it is a defense-in-depth gap.

**Recommendation**: Add `if (disabled) return;` at the top of `handleKeyDown`.

## 5. Code Quality Observations (Non-Bug)

### OBS-001: `useCallback` Dependencies Include `value` in `handleKeyDown`

**Location**: `C:\Velocity\src\components\editor\InputEditor.tsx`, line 43

```typescript
const handleKeyDown = useCallback(
  (e: React.KeyboardEvent<HTMLTextAreaElement>) => {
    if (e.key === 'Enter' && !e.shiftKey) {
      e.preventDefault();
      onSubmit(value);
    } else if (e.key === 'Tab') {
      // ...uses value...
    }
  },
  [value, onSubmit, onChange],
);
```

The `handleKeyDown` callback is recreated on every keystroke because `value` is in the dependency array and `value` changes on every character typed. This means a new function reference is passed to the textarea's `onKeyDown` on every render. This is not a performance issue (the textarea re-renders anyway due to `value` changing), but it negates the purpose of `useCallback`.

**Assessment**: Not a bug. The `useCallback` is necessary for correctness (needs current `value` for `onSubmit(value)` and Tab insertion). The dependency is correct.

## 6. Security Assessment

### Positive Findings

1. **No `dangerouslySetInnerHTML`** -- Token rendering uses safe React children (`{token.value}` inside `<span>` elements). No XSS vector through the tokenizer or overlay.
2. **Tokenizer is display-only** -- The tokenizer does NOT affect what is sent to the shell. The raw `value` string is passed to `onSubmit` unchanged. There is no risk of the tokenizer altering command semantics.
3. **No shell command interpolation** -- The input value is passed directly to `writeToSession` via Tauri IPC. No string interpolation or template literal injection.
4. **`aria-hidden="true"` on overlay** -- The `<pre>` overlay is correctly hidden from screen readers, preventing duplicate content announcements.
5. **No new IPC surface** -- The input editor is purely frontend. No new Tauri commands, no new Rust code.

### Security Concern: Multi-line Input Bypass

The old `<input type="text">` naturally prevented multi-line input. The new textarea allows arbitrary multi-line content. If a user pastes content from an untrusted source (e.g., a webpage with hidden newlines), the multi-line input would send those lines to the shell. This is standard terminal behavior (paste-jacking is a known terminal security concern), but it is worth noting that the attack surface for paste-jacking has increased with the switch to textarea.

**Assessment**: This is inherent to any multi-line terminal input and is not specific to this implementation. No action needed, but consider adding a "bracketed paste" mode or paste confirmation in a future iteration.
