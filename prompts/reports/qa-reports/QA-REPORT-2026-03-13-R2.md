## 4. New Bugs Found

### BUG-033: Closing a Tab Does Not Guarantee Synchronous PTY Session Cleanup

**Severity**: Medium
**Location**: `C:\Velocity\src\components\layout\TabManager.tsx`, line 37-51; `C:\Velocity\src\components\Terminal.tsx`, lines 185-198

**Description**: When a tab is closed via `handleCloseTab`, the tab is removed from the `tabs` array, which causes React to unmount the `<Terminal />` component. The Terminal's `useEffect` cleanup function (line 188) calls `closeSession(sessionIdRef.current).catch(() => {})` to clean up the PTY session.

This cleanup is **asynchronous and fire-and-forget**:
1. `closeSession` is an async IPC call to the Rust backend
2. The `.catch(() => {})` swallows all errors silently
3. There is no feedback to the TabManager that cleanup succeeded or failed

If `closeSession` fails (e.g., the Mutex is poisoned, the session was already closed by a race condition, or the Rust backend is overloaded), the PTY process continues running as an orphan -- consuming a session slot (of the 20 max) and a system process handle. Since the frontend tab is gone, the user has no way to close the orphaned session.

Additionally, the Rust `close_session` method (pty/mod.rs line 384-407) does a blocking `thread::sleep(100ms)` and `child.wait()` inside `spawn_blocking`. If many tabs are closed rapidly, these blocking calls pile up on the Tokio thread pool.

**Impact**: Under normal single-tab-close usage, this works fine. Under rapid multi-tab closure (e.g., closing 5 tabs quickly), orphaned sessions may accumulate. The user would see "Maximum session limit (20) reached" eventually with no way to free the orphaned slots.

**Recommendation**:
1. Add error logging (not just swallowing) on cleanup failure
2. Consider adding a `list_sessions` command so the frontend can audit/clean orphaned sessions
3. For the `close_session` Rust path, consider reducing the 100ms sleep or making it configurable

### BUG-034: No Frontend Enforcement of MAX_SESSIONS Tab Limit

**Severity**: Medium
**Location**: `C:\Velocity\src\components\layout\TabManager.tsx`, line 24-33

**Description**: The `handleNewTab` callback creates a new tab unconditionally. There is no check against the backend's `MAX_SESSIONS = 20` limit. When the user creates the 21st tab, the `<Terminal />` component will mount and call `createSession`, which will fail with "Maximum session limit (20) reached". The Terminal component handles this by showing an error in the welcome block:

```typescript
const errorBlock = createBlock('', shell);
errorBlock.output = `[Failed to create session: ${err}]`;
```

However, the user sees a tab with a broken terminal displaying an error message. There is no indication in the TabBar that this tab is in a failed state, no prevention of creating more broken tabs, and no way to know (short of reading the error) that the session limit was hit.

**Impact**: A user who opens many tabs will encounter a confusing error state. Each additional tab beyond 20 creates a broken tab that occupies UI space. The user must manually close these broken tabs.

**Recommendation**:
1. Add a `getSessionCount` command to the Rust backend (or have `createSession` return the count)
2. Disable the `+` button and Ctrl+T when approaching the limit
3. Show a toast/notification when the limit is reached
4. At minimum, pass the error from Terminal up to TabManager so it can remove the failed tab automatically

### BUG-035: `autoFocus` on All Terminal InputEditors Causes Focus Fights

**Severity**: Low
**Location**: `C:\Velocity\src\components\editor\InputEditor.tsx`, line 62

**Description**: The `<textarea>` in `InputEditor` has `autoFocus` set unconditionally. When multiple tabs exist, ALL Terminal instances (including hidden ones) have their InputEditor rendered with `autoFocus`. When a new tab is created:

1. React renders the new `<Terminal />` component
2. The new Terminal's InputEditor textarea has `autoFocus`
3. The browser focuses the new textarea

This actually works correctly because the new tab IS the active tab, so the focused element is in the visible panel. However, on initial mount with React StrictMode (double-mount), or when switching tabs causes a re-render, the `autoFocus` on hidden tab textareas is a latent issue -- the browser will focus whichever textarea was last rendered/mounted, which may be in a hidden panel.

In practice, `autoFocus` only fires on initial mount, not on re-renders, so this is unlikely to manifest. But if a tab's Terminal remounts for any reason (e.g., error recovery in a future version), focus could jump to a hidden tab.

**Impact**: Minimal in current implementation. Latent risk for future changes.

**Recommendation**: Pass an `isFocused` or `isActive` prop to Terminal/InputEditor based on whether the tab is active, and only set `autoFocus` (or use a `useEffect` with `focus()`) on the active tab's textarea.

### BUG-036: Closing the Active Tab When It Is the First Tab Selects Index 0 (Self)

**Severity**: Low (Actually Not a Bug -- See Analysis)
**Location**: `C:\Velocity\src\components\layout\TabManager.tsx`, lines 40-48

**Description**: The close logic is:
```typescript
const index = prev.findIndex((t) => t.id === tabId);
const newTabs = prev.filter((t) => t.id !== tabId);
if (tabId === activeTabIdRef.current) {
  const newActiveIndex = index > 0 ? index - 1 : 0;
  updateActiveTabId(newTabs[newActiveIndex].id);
}
```

When closing the first tab (index 0) while it is active:
- `index = 0`, so `newActiveIndex = 0` (the `index > 0` check fails)
- `newTabs[0]` is the SECOND original tab (since the first was filtered out)
- This correctly selects the next tab after removal

**Analysis**: This is actually correct behavior. The `newTabs` array has already had the closed tab removed, so `newTabs[0]` is the new first tab (originally the second tab). No bug here.

**Status**: NOT A BUG -- verified through code analysis. Closing for record.

### BUG-038: Keyboard Shortcuts Not Scoped to Application Window

**Severity**: Low
**Location**: `C:\Velocity\src\components\layout\TabManager.tsx`, lines 57-71

**Description**: The keyboard shortcuts (Ctrl+T, Ctrl+W) are registered on `document.addEventListener('keydown', ...)`. In a Tauri application, this is fine because the document IS the application window. However:

1. The shortcuts call `e.preventDefault()` which suppresses the browser-default behavior. In the Tauri webview, Ctrl+T and Ctrl+W do not have default browser behavior (no browser tab bar), so `preventDefault` is a no-op but harmless.

2. If in the future the application embeds iframes or web content (unlikely for a terminal), the shortcuts would fire on events bubbling from those elements.

3. The Ctrl+W shortcut always fires, even when there is only 1 tab. The `handleCloseTab` function safely prevents closing the last tab (`if (prev.length <= 1) return prev`), so no harm is done, but the `e.preventDefault()` still fires. In a browser context (development), this would prevent the browser from closing the tab, which can be annoying during development with `npm run tauri dev` (which opens in a browser-like webview).

**Impact**: Minimal. The shortcuts work correctly in the Tauri context.

**Recommendation**: Consider only calling `e.preventDefault()` for Ctrl+W when there are multiple tabs (matching the visual hiding of the close button). This avoids suppressing the browser close-tab behavior during development when only one tab exists.

## 5. Code Quality Observations (Non-Bug)

### OBS-004: Tab Interface Has Clean Component Separation

The architecture cleanly separates concerns:
- `TabBar` is a pure presentational component (no state, no side effects, receives all data via props)
- `TabManager` manages tab state and delegates all rendering to TabBar and Terminal
- `Terminal` is fully self-contained and unaware of the tab system

This means Terminal requires zero modifications to support tabs, which is excellent. The `display: none` approach preserves React component state without any special plumbing.

### OBS-005: `activeTabIdRef` Pattern is Sound

The `activeTabIdRef` + `updateActiveTabId` pattern (fixed in R2 per code review F-01) correctly prevents stale closures in `handleCloseTab` and the keyboard shortcut `useEffect`. This is the recommended React pattern for accessing current state in callbacks that should not re-register on state changes.

### OBS-006: `Tab.shellType` is Stored but Not Used for Tab Title

The `Tab` interface includes `shellType` but the title is always "Terminal N" (global counter). The task spec mentions that future work could use shell-specific titles ("PowerShell 1", "CMD 1"). The `shellType` field is correctly present for this future enhancement.

### OBS-007: CSS `.tab-panel` Missing Explicit `display` Property

As noted in code review F-03 (accepted as nit), the `.tab-panel` CSS class sets `flex: 1` and `flex-direction: column` but does not set `display: flex`. The inline style `display: flex` on the active panel works because the inline style takes precedence. The hidden panels get `display: none` which overrides everything. So this works correctly at runtime, but the CSS class alone is technically incomplete (flex properties have no effect without `display: flex`).

## 6. Security Assessment

### Positive Findings

1. **No new IPC surface**: The tab system is purely frontend state management. No new Tauri commands were added. Each tab's Terminal uses the existing `create_session`, `start_reading`, `write_to_session`, `close_session` commands.

2. **Tab IDs use `crypto.randomUUID()`**: Non-guessable, consistent with block IDs and session IDs. No sequential/predictable IDs that could be enumerated.

3. **Session cleanup on unmount**: The Terminal's `useEffect` cleanup properly closes sessions when tabs are removed. The cleanup path was already reviewed and hardened in previous iterations (StrictMode double-mount handling, startSessionIdRef cancellation guard).

4. **No cross-tab data leakage**: Each Terminal maintains its own `sessionIdRef`, `blocks`, `input`, and event listeners. Output events are scoped to `pty:output:${sid}` -- a tab can only receive events for its own session ID.

5. **Backend session limit enforced**: The Rust `SessionManager` enforces `MAX_SESSIONS = 20` regardless of how many tabs the frontend creates.

### Security Concern: Session Exhaustion via Tab Spam

A user (or automated script interacting with the Tauri window) could rapidly create tabs via Ctrl+T, exhausting the 20-session limit. Each tab spawns a system process (powershell.exe). While 20 is a reasonable limit, there is no rate limiting on tab creation. A malicious webpage embedded in the webview (if that were ever supported) could spam Ctrl+T events.

**Assessment**: Low risk in current architecture (Tauri webview only loads local content, no remote pages). The 20-session backend limit provides adequate protection. No action needed.
